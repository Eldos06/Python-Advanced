# Генераторы и итераторы

Прогрессия: протокол итератора вручную (`__iter__`/`__next__`) → генераторы (`yield`) как более простой способ получить
то же самое → генераторы как корутины (`send`/`throw`/`yield from`) для двустороннего обмена данными.

## 1. Итератор vs итерируемый объект — [[5.Generators_and_Iterators/1.generators_iterators.py|1.generators_iterators.py]]

Ключевое наблюдение файла: список - итерируемый (`__iter__`), но не итератор сам по себе. При работе с файлом
`for line in f` и ручной `next(f)` **используют один и тот же курсор**:

```python
with open(file, "r") as f:
    for line in f:
        print(repr(line))
        if "7" in line:
            break
    print("One more line:")
    print(repr(next(f)))  # продолжает СО СЛЕДУЮЩЕЙ строки после break, а не с начала файла
```

Файл - это итератор с состоянием: `for` не "перезапускает" его, а просто вызывает `next()` по кругу, поэтому
позиция чтения сохраняется между `for` и ручным `next()`.

## 2. Свой итератор через `__iter__`/`__next__` — [[5.Generators_and_Iterators/3.create_itaration.py|3.create_itaration.py]]

```python
class Range:
    def __init__(self, start, stop=None, step=1):
        ...
        self._current = self.start

    def __iter__(self):
        self._current = self.start  # сброс состояния при новом for
        return self

    def __next__(self):
        if self._current < self.stop:
            current = self._current
            self._current += self.step
            return current
        raise StopIteration
```

`StopIteration` - это единственный сигнал для `for`, что элементы кончились; без него цикл будет пытаться дергать
`__next__` бесконечно.

> [!WARNING] Итератор, который не сбрасывается в `__iter__`
> Если `__iter__` не переустанавливает `self._current`, второй `for i in r:` по тому же объекту `r` ничего не выведет -
> состояние уже "исчерпано" предыдущим циклом. Важно różличать: **итератор** хранит позицию и одноразовый,
> **итерируемый объект** (у которого `__iter__` каждый раз создаёт новый итератор) можно обходить многократно.

Дальше на этой идее строятся `fibIter`/`Count`/`Cycle`/`CycleN` (наследники `typing.Iterator`, который просто требует
`__next__`) - циклический перебор последовательности (`Cycle`) и та же идея с ограничением числа повторов (`CycleN`,
кидает `StopIteration` после `repeat` кругов). `itertools.count`/`cycle`/`repeat` в стандартной библиотеке - готовые
аналоги этих же классов.

## 3. Генераторы — [[5.Generators_and_Iterators/2.create_generators.ipynb|2.create_generators.ipynb]]

Та же идея, что и `Range`/`fibIter`, но без ручного класса - функция с `yield` сама становится итератором:

```python
def fib():
    a, b = 0, 1
    while True:
        yield a
        a, b = b, a + b
```

Ключевые наблюдения из ноутбука:
- Генератор **не выполняется**, пока не вызван `next()` - `func_g()` возвращает объект-генератор, тело функции
  начинает исполняться только на первом `next(g)`.
- Генератор **ленивый** и **экономит память**: `large_sequence(1_000_000)` не создаёт список на миллион элементов -
  значения считаются по требованию.
- `next(gen, default)` - безопасный способ получить значение или запасное (`default`), не ловя `StopIteration`.
- `gen.close()` останавливает генератор досрочно (это бросает `GeneratorExit` внутрь - можно перехватить в `finally`
  для очистки ресурсов).

## 4. Генераторы как корутины: `send`/`throw`/`yield from` — [[5.Generators_and_Iterators/4.coroutines.py|4.coroutines.py]]

`yield` - это выражение, а не только оператор: `x = yield` умеет **принимать** значение снаружи через `.send(value)`:

```python
def calc():
    while True:
        task = yield
        result = eval(task)
        print(f"{task} = {result}")

clc = calc()
next(clc)          # обязательный "прайминг" - довести генератор до первого yield
clc.send("2 + 3")  # выведет: 2 + 3 = 5
```

> [!NOTE] Зачем нужен `next()` перед первым `send()`
> `send(value)` кладёт `value` в место, где генератор сейчас стоит на `yield`. Если генератор ещё не начал
> выполняться (не дошёл до первого `yield`), слать в него нечего - Python выбросит `TypeError`. Поэтому первый
> вызов всегда `next(gen)` (или `gen.send(None)`), и только потом настоящие `send(value)`.

### Конвейер корутин (`send` пробрасывает значение дальше по цепочке)

```python
s = summator()
next(s)
sq = square(s)     # square шлёт результат в s
next(sq)
f = even_filter(sq)  # even_filter шлёт чётные числа в sq
next(f)

f.send(2); f.send(3); f.send(4)
f.send(None)  # закрывает цепочку -> StopIteration(20)  (4**2 + 4**2? нет: 2**2 + 4**2 = 4+16=20)
```

Каждая корутина в цепочке - фильтр/трансформация, следующая за `yield` пересылкой `target.send(...)`. Останов и
возврат финального значения идёт через `except`/`return` и `StopIteration.value`.

### `yield from` — делегирование другому генератору

```python
def collect_counters(counters: list):
    while True:
        counter = yield from add_count()  # передаёт send()'ы во вложенный генератор,
        counters.append(counter)          # и получает его return-значение как результат yield from
```

`yield from add_count()` - не просто ярлык для цикла `for`, а полноценное делегирование: все `.send()`/`.throw()`,
адресованные `collect_counters`, идут прямиком во вложенный `add_count()`, а когда тот завершается через `return`,
это возвращаемое значение становится результатом самого выражения `yield from`.
