# Аннотации типов (typing)

От простых аннотаций (`x: int`) к дженерикам (`Generic[T]`, `TypeVar`), `Self`/`Type[T]` и типизации генераторов.

## 1. Дженерики: `Generic[T]` — [[9.AnotationType/1.AnotationType/container.py|container.py]]

```python
T = TypeVar('T')

class Container(Generic[T]):
    def __init__(self, data: T) -> None:
        self.data = data
```

`Container` типобезопасно оборачивает данные **любого одного** типа `T` - `Container[str]` и `Container[int]` для
type-checker'а (mypy) это разные специализации одного generic-класса:

```python
data_int_or_str = Container[str | int](234)  # явная параметризация generic-класса
```

## 2. `TypeVar`, `Union`/`|`, `Iterable`, `Generator[...]` — [[9.AnotationType/1.AnotationType/main.py|main.py]]

```python
NumInt = TypeVar("NumInt", bound=float)   # bound: T должен быть float ИЛИ его подкласс (int сюда не подходит по MRO)
T = TypeVar("T")                          # без bound - T это "какой угодно один и тот же тип"

def add(a: T, b: T) -> T:                 # a и b должны быть ОДНОГО типа, и результат тоже
    return a + b
```

```python
assert float | str == Union[float, str]   # современный синтаксис (PEP 604) - то же самое, что typing.Union
```

Типизация генератора — три параметра описывают весь протокол `send`/`yield`/`return`:

```python
def add_nums() -> Generator[int, NumInt, tuple[int, list[NumInt]]]:
    #                        ^yield    ^send    ^return
    all_inputs: list[NumInt] = []
    total = 0
    while True:
        num: NumInt = yield len(all_inputs)
        if num is None:
            return total, all_inputs
        all_inputs.append(num)
        total += num
```

`Generator[YieldType, SendType, ReturnType]` - `YieldType` это то, что отдаёт `yield`, `SendType` - то, что
принимает `.send(...)`, `ReturnType` - то, что попадёт в `StopIteration.value` при `return`.

> [!NOTE] Аннотации - это подсказки, не runtime-проверки
> `T = TypeVar("T")` в `add(a: T, b: T) -> T` не мешает вызвать `add(["foo"], ["bar", "eggs"])` или
> `add([1, 1, ""], ["bar", "eggs"])` в `main()` - Python не проверяет типы во время выполнения, это только сигнал для
> mypy/IDE. Несовпадение `T` у `a` и `b` во втором вызове mypy пометил бы ошибкой, а сам Python выполнит без вопросов.

## 3. `Self`, `Type[T]`, classmethod-фабрика — [[9.AnotationType/1.AnotationType/models.py|models.py]]

```python
class BaseModel:
    ModelType = TypeVar('ModelType', bound='BaseModel')

    @classmethod
    def create(cls: Type[ModelType], foo: str | float) -> ModelType:
        bar = foo * 3
        return cls(foo=foo, bar=bar)   # cls - это конкретный подкласс, не обязательно BaseModel

    def update(self) -> Self:
        return self
```

`cls: Type[ModelType]` в паре с `-> ModelType` - стандартный паттерн для фабричных `classmethod`, чтобы
`User.create(...)` возвращал тип `User`, а не жёстко `BaseModel`, даже если тело метода описано в базовом классе.
`Self` (PEP 673) делает то же самое для обычных методов проще - `update(self) -> Self` говорит "возвращает объект
того же класса, что и `self`", без ручного `TypeVar`.

## 4. `Queue[T]` — [[9.AnotationType/1.AnotationType/practice.ipynb|practice.ipynb]]

```python
class Queue(Generic[T]):
    def __init__(self) -> None:
        self._items: list[T] = []

    def enqueue(self, item: T) -> None: ...
    def dequeue(self) -> T: ...

str_queue = Queue[str]()   # T зафиксирован как str для этого конкретного экземпляра
str_queue.enqueue("Первый")
# str_queue.enqueue(123)  # рантайм это пропустит, но mypy/PyCharm подсветят как ошибку типа
```

> [!WARNING] `next()` вместо `send()` "теряет" отправленные данные
> В том же ноутбуке `for i in range(10): print(next(f))` над `add_nums()` падает с `StopIteration` уже на втором
> вызове: `next(f)` эквивалентно `f.send(None)`, а генератор трактует `None` как сигнал завершения
> (`if num is None: return total, all_inputs`). Чтобы реально копить числа, нужно звать `f.send(число)`, как в
> [[9.AnotationType/1.AnotationType/main.py|main.py]] (`process_nums`), а не голый `next()` в цикле.
