# Функциональное программирование в Python

Три темы: пределы рекурсии и хвостовая рекурсия, `functools.reduce`/`operator`, и производительность `map`/`filter`
по сравнению с list comprehension.

## 1. Рекурсия и её пределы — [[4.functional-programming-python/2.py|2.py]]

```python
def fag(n):
    if n <= 2:
        return n
    return n * fag(n-1)
```

Наивная рекурсивная факториал-подобная функция упирается в `sys.getrecursionlimit()` (по умолчанию 1000) - для больших
`n` (например 1558) Python выбросит `RecursionError`, поэтому в файле `setrecursionlimit(2001)` увеличивает лимит.

```python
def fact(k):
    result = 1
    for i in range(1, k+1):
        result *= i
    return result
```

Итеративная версия того же факториала не имеет проблем с лимитом рекурсии вообще - это стандартный обход "просто
перепиши рекурсию в цикл".

> [!NOTE] Хвостовая рекурсия в Python не оптимизируется
> Закомментированный блок с `TatilCall`/`tail_recursion`/`tail` - это trampoline-паттерн: вместо реального рекурсивного
> вызова функция возвращает объект-"обещание" (`TatilCall`), а внешний `wrapper` в цикле `while` разворачивает его сам.
> Так эмулируют tail-call optimization, потому что CPython, в отличие от некоторых других языков, её не делает —
> глубокая рекурсия всегда тратит стек, вне зависимости от формы записи.

### `reduce` vs `lambda` в `reduce` — сравнение через `timeit`

```python
timeit.timeit(stmt="reduce(mul, range(1, 1559))", setup="from functools import reduce; from operator import mul", number=1000)
timeit.timeit(stmt="reduce(lambda a, b: a * b, range(1, 1559))", setup="from functools import reduce;", number=1000)
```

`operator.mul` как функция для `reduce` немного быстрее эквивалентной `lambda a, b: a * b` - меньше накладных расходов
на вызов Python-функции.

## 2. `map`/`filter` vs list comprehension — [[4.functional-programming-python/Comparison of helpers map, filter with comprehensions. Lambda functions.py|Comparison...py]]

Один и тот же результат тремя способами:

```python
[abs(num) for num in someNum]                 # comprehension
list(map(abs, someNum))                       # map с именованной функцией
(num ** 2 for num in someNum)                 # генератор
```

`map` с несколькими итерируемыми объектами применяет функцию поэлементно, как `zip` + вызов:

```python
list(map(power, someNum, sumPowers))          # power(someNum[i], sumPowers[i]) для каждого i
list(map(add, first, second))                 # эквивалент [a+b for a, b in zip(first, second)]
```

Файл гоняет `timeit` на конкурирующих вариантах (`map(str.title, names)` vs `[name.title() for name in names]` vs
`map(lambda ...)`), чтобы сравнить реальную скорость - вывод обычно такой:

> [!NOTE] Что обычно быстрее
> `map` с **встроенной** функцией (`str.title`, `abs`, `operator.add`) часто быстрее comprehension на больших данных,
> потому что вызов идёт на уровне C без байткода Python на каждой итерации. Но `map` с `lambda` теряет это преимущество -
> вызов пользовательской функции всё равно проходит через интерпретатор, и часто оказывается медленнее comprehension.

`filter` аналогично сравнивается с comprehension-фильтрацией:

```python
list(filter(isNotNegative, someNum))          # filter с функцией
list(filter(lambda n: n >= 0, someNum))       # filter с лямбдой
[n for n in someNum if n >= 0]                # comprehension
```
