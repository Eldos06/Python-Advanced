# Декораторы классов, `*args`/`**kwargs` — конспект

Декоратор класса — это то же самое, что декоратор функции, только на вход и (обычно) на выход подаётся класс, а не функция. Разбор трёх паттернов из папки [[2.Углубленная работа с декораторами/2_Классы-декораторы|2_Классы-декораторы]] и упражнений на `*args`/`**kwargs`.

## 1. Паттерн "мутировать класс на месте"

[[2.Углубленная работа с декораторами/2_Классы-декораторы/1. Простой декоратор для класса.py|1. Простой декоратор для класса.py]]:

```python
def add_description(cls):
    cls.description = "This is a class"
    return cls

@add_description
class MyClass:
    id = int
    name = str
```

Декоратор просто дописывает атрибут в сам класс и возвращает **тот же объект класса**. Это самый безопасный вид декоратора класса: `MyClass` после декорирования остаётся классом `MyClass`, `isinstance`/`issubclass` работают как обычно.

## 2. Паттерн "вернуть не класс, а функцию" — скрытая ловушка

[[2.Углубленная работа с декораторами/2_Классы-декораторы/2. Декоратор класса для регистрации экземпляров.py|2. Декоратор класса для регистрации экземпляров.py]]:

```python
instances = {}

def track_instances(cls):
    cls.__models_registry__ = instances
    def wrapper(*args, **kwargs):
        instance = cls(*args, **kwargs)
        instances[cls.__name__] = instances.get(cls.__name__, []) + [instance]
        return instance
    return wrapper

@track_instances
class Person:
    pass
```

> [!WARNING] После декорирования `Person` — это уже не класс
> `track_instances` возвращает `wrapper` — обычную функцию, а не класс `cls`. Имя `Person` в модуле теперь ссылается на функцию `wrapper`, которая при вызове создаёт экземпляр *настоящего*, скрытого в замыкании класса. Следствия:
> - `Person()` по-прежнему работает и создаёт объект — но `isinstance(person1, Person)` **упадёт с `TypeError`**, т.к. `Person` больше не класс, а функция;
> - `Person.__models_registry__` работать не будет — атрибут был поставлен на исходный `cls`, а не на `wrapper`.
>
> Реестр `instances` в этом файле — общий модульный словарь, поэтому сам трекинг экземпляров (`print(instances)`) отрабатывает верно; проблема именно в подмене самого класса функцией.

## 3. Паттерн "вернуть подкласс" — правильный способ добавить поведение

[[2.Углубленная работа с декораторами/2_Классы-декораторы/practice1.py|practice1.py]] и [[2.Углубленная работа с декораторами/2_Классы-декораторы/practice2.py|practice2.py]] решают ту же задачу (добавить классу метод/атрибут) через наследование — и остаются классами:

```python
def add_prefix(prefix):
    def class_decorator(class_to_decorate):
        class DecoratedClass(class_to_decorate):
            def __init__(self, *args, **kwargs):
                super().__init__(*args, **kwargs)
                self.prefix = prefix
            def get_prefixed_name(self):
                return self.prefix + self.name
        return DecoratedClass
    return class_decorator

@add_prefix("Mr. ")
class Person:
    def __init__(self, name):
        self.name = name

person = Person("John")
print(person.get_prefixed_name())  # Mr. John
```

`add_prefix(prefix)` — внешняя фабрика (даёт параметризацию, как `trace(sep=...)` в [[Вложенные декораторы]]), `class_decorator` — сам декоратор класса. `DecoratedClass` наследуется от исходного класса и обязательно вызывает `super().__init__(*args, **kwargs)`, иначе атрибуты родителя (`self.name`) не будут проинициализированы. Тот же приём использует `main.py` (`add_new_method`), добавляя классу метод `power` через подмешивание подкласса.

> [!NOTE] Почему это лучше, чем пункт 2
> `DecoratedClass` — настоящий класс, наследник исходного. `isinstance(person, Person)` вернёт `True` (т.к. `DecoratedClass` — подкласс `Person`), а не упадёт с ошибкой.

Механика `super()` в подклассах разобрана отдельно в [[2.Углубленная работа с декораторами/2_Классы-декораторы/super vs self.py|super vs self.py]]: `super().__init__(name)` вызывает `__init__` родителя вместо дублирования кода через `self.name = name`, а `super().speak()` переиспользует поведение родителя перед тем, как дополнить его в наследнике.

## 4. `handle_errors` — декоратор-обработчик ошибок

[[2.Углубленная работа с декораторами/2_Классы-декораторы/handle_errors.py|handle_errors.py]] — обычный (не классовый) декоратор функции, гасящий конкретные исключения:

```python
def handle_errors(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        try:
            return func(*args, **kwargs)
        except (ValueError, ZeroDivisionError, KeyError) as e:
            print(f"[{type(e).__name__}] - {e}")
        except Exception as e:
            print(f"[UNEXPECTED ERROR] - {type(e).__name__}: {e}")
    return wrapper
```

Сначала перехватываются три ожидаемых типа ошибок с отдельным сообщением, затем — любой другой `Exception` как "неожиданная" ошибка. Такой же двухуровневый `try/except` пригодится при обработке исключений — см. [[Обработка исключений]].

## 5. `*args` и `**kwargs`

Отработано в папке `args vs kwargs/`.

| | `*args` | `**kwargs` |
|---|---|---|
| Что собирает | позиционные аргументы в `tuple` | именованные аргументы в `dict` |
| [[2.Углубленная работа с декораторами/2_Классы-декораторы/args vs kwargs/practice1.py|practice1.py]] | `multip(*args)` — произведение всех чисел | — |
| [[2.Углубленная работа с декораторами/2_Классы-декораторы/args vs kwargs/practice2.py|practice2.py]] | — | `person_details(**kws)`, `None` заменяется на `"Unknown"` |
| [[2.Углубленная работа с декораторами/2_Классы-декораторы/args vs kwargs/practice3.py|practice3.py]] | `Student(*a, **kw)` — комбинация: `a[0]` = имя, `a[1:]` = предметы, `kw.get("age", ...)` | |

```python
class Student:
    def __init__(self, *a, **kw):
        self.a = a
        self.kw = kw
    def display_info(self):
        name = self.a[0]
        age = self.kw.get("age", "Unknown")
        subjects = self.a[1:]
        print(f"Name: {name}, Age: {age}, Subjects: {subjects}")
```

[[2.Углубленная работа с декораторами/2_Классы-декораторы/args vs kwargs/practice4.py|practice4.py]] повторяет паттерн `functools.wraps` из [[Вложенные декораторы]] (декоратор `hello`, оборачивающий `name()`).

> [!WARNING] Опечатка в [[2.Углубленная работа с декораторами/2_Классы-декораторы/args vs kwargs/practice5.py|practice5.py]]
> ```python
> def showInfo(self):
>     return f"name: {self.name} breed: {self.breed}  age: {self.breed}"
> ```
> Для `age` по ошибке второй раз подставлено `self.breed` вместо `self.age` — `tuz.showInfo()` печатает возраст, равный породе-строке, вместо числа `32`.

## Ещё один пример "вернуть подкласс" — [[2.Углубленная работа с декораторами/2_Классы-декораторы/main.py|main.py]]

Тот же безопасный паттерн из пункта 3 (наследник вместо мутации/подмены класса), в финальной (не закомментированной) части файла:

```python
def add_new_method(class_to_decorate):
    class DecoratedClass(class_to_decorate):
        def power(self, a, b):
            self.a, self.b = a, b
            return f"power: {a ** b}"
    return DecoratedClass

@add_new_method
class MyClass:
    def multi(self, a, b):
        return f"multipli is {a * b}"
```

`numbers = MyClass()` умеет и `multi` (из исходного класса), и `power` (добавленный декоратором подкласс) - оба метода
доступны на одном объекте. Остальная часть файла - черновики-эксперименты с реестрами классов (`register_model`,
`ModelsRegistry`) и кэширующим декоратором `Cached`/`History`, закомментированные один за другим по мере проработки темы.
