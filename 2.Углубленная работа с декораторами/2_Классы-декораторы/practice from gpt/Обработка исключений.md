# Обработка исключений — конспект

Две небольшие практики на `try/except`, кастомные исключения и черновой retry-декоратор до его "чистого" переоформления в [[Retry decorator]].

## 1. `weather/` — черновик retry-декоратора

[[2.Углубленная работа с декораторами/2_Классы-декораторы/practice from gpt/weather/WeatherDataError.py|WeatherDataError.py]] и [[2.Углубленная работа с декораторами/2_Классы-декораторы/practice from gpt/weather/copyweather.py|copyweather.py]] — почти дублирующиеся черновики одной идеи:

```python
class WeatherDataError(Exception):
    pass

def retry(max_retries=3):
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            for attempt in range(1, max_retries + 1):
                try:
                    return func(*args, **kwargs)
                except WeatherDataError as e:
                    print(f"Attempt #{attempt} failed: {e}")
                    if attempt == max_retries:
                        raise
                    time.sleep(1)
        return wrapper
    return decorator
```

Здесь `get_weather()` случайно (`random.choice`) бросает `WeatherDataError` с текстом `"API failed"`/`"Invalid data"`, а декоратор гасит и повторяет попытку, пока не кончится лимит.

> [!NOTE] Это тот же `retry`, что и в [[Retry decorator]], только в зачаточном виде
> Отличия от "боевой" версии в `practice from gpt/retry/`: захардкожен один тип исключения (`WeatherDataError`) вместо настраиваемого `exceptions=`, нет экспоненциального роста задержки (всегда `time.sleep(1)`), нет типизации через `ParamSpec`/`TypeVar`, и результат ретраев не логируется — печатается через `print`.

## 2. `Исключение/` — точечные упражнения на `try/except`

### Порядок `except` имеет значение

[[2.Углубленная работа с декораторами/2_Классы-декораторы/practice from gpt/Исключение/main.py|main.py]]:

```python
def main():
    data = {'surname': 'Suleimenov', 'name': 'Yeldos'}
    try:
        print(sdf)              # <- имя не определено!
        print(data['surname'])
        print(data['name'])
    except KeyError:
        sum = 1 + 1
        print(sum)
    except Exception:
        print("oops")
    finally:
        print("finished")
```

> [!WARNING] `print(sdf)` роняет `NameError`, а не `KeyError`
> Обращения к `data['surname']`/`data['name']` вообще не выполняются — падение происходит раньше, на `print(sdf)` (переменная `sdf` нигде не объявлена). `NameError` не совпадает с первым `except KeyError`, поэтому его ловит именно широкий `except Exception`, печатая `"oops"`. `finally` отрабатывает в любом случае. Это наглядно показывает: except-блоки проверяются по порядку сверху вниз, и срабатывает первый подходящий по типу.

### Несовпадение типа исключения

[[2.Углубленная работа с декораторами/2_Классы-декораторы/practice from gpt/Исключение/dic.py|dic.py]]:

```python
def get_value(data: dict, key: str):
    if key not in data:
        raise KeyError(f'KeyError: Key not found: {key}')
    return data[key]

try:
    print(get_value({"name": "Baha"}, "hehe"))
except ValueError as e:
    print("Error:", e)
```

> [!WARNING] `except ValueError` не поймает `KeyError`
> `get_value` явно бросает `KeyError`, а обработчик ловит `ValueError`. Исключение не перехватывается и приведёт к падению программы с трейсбеком — классическая ошибка "перепутал тип исключения в `except`".

### Раздельные проверки одного набора данных

[[2.Углубленная работа с декораторами/2_Классы-декораторы/practice from gpt/Исключение/process_data.py|process_data.py]] (и почти идентичный по структуре
[[2.Углубленная работа с декораторами/2_Классы-декораторы/practice from gpt/Исключение/calculator.py|calculator.py]] — та же цепочка
`KeyError`/`ValueError`/`ZeroDivisionError`, только результат в конце ещё проверяется на отрицательность и идёт в `math.sqrt`) показывают чистый способ валидации по цепочке:

```python
def process_data(data: dict):
    if "a" not in data or "b" not in data:
        raise KeyError("Key 'a' or 'b' not found")
    a, b = data["a"], data["b"]
    if not isinstance(a, int) or not isinstance(b, int):
        raise ValueError("Inputs must be integers")
    if b == 0:
        raise ZeroDivisionError("Cannot divide by zero")
    return a / b
```

На вызывающей стороне достаточно одного универсального перехвата, т.к. каждая проверка бросает свой конкретный тип:

```python
for case in test_cases:
    try:
        result = process_data(case)
    except Exception as e:
        print(f"{type(e).__name__} - {e}")
```

> [!NOTE] Отличие от `avg.py`
> В [[2.Углубленная работа с декораторами/2_Классы-декораторы/practice from gpt/Исключение/avg.py|avg.py]] проверка `if not isinstance(a, int)` внутри `get_average` фактически недостижима: значения `a`/`b` уже приходят из `int(input(...))`, то есть либо гарантированно `int`, либо `int()` сам бросит `ValueError` раньше, чем управление дойдёт до тела `get_average`.

### Файлы и жёстко прописанные пути

[[2.Углубленная работа с декораторами/2_Классы-декораторы/practice from gpt/Исключение/read_f.py|read_f.py]] — аккуратная цепочка обработки при чтении файла:

```python
try:
    with open(<путь>, 'r', encoding='utf-8') as text_file:
        content = text_file.read()
        if not content:
            raise ValueError("Файл пустой")
        return content
except FileNotFoundError:
    raise FileNotFoundError("Файл не найден")
except ValueError as ve:
    raise ve
except Exception as e:
    raise Exception(f"Неизвестная ошибка: {e}")
```

> [!WARNING] Абсолютный путь захардкожен под конкретную машину
> Путь к [[2.Углубленная работа с декораторами/2_Классы-декораторы/practice from gpt/Исключение/file.txt|file.txt]] в `read_f.py` начинается с `C:/Users/Acer Nitro 5/Desktop/...` — при переносе проекта на другой компьютер (как и произошло с текущим репозиторием) файл по такому пути не существует, и сценарий всегда падает в ветку `except FileNotFoundError`. Пути к файлам стоит собирать относительно расположения скрипта (`Path(__file__).parent / "file.txt"`), а не прописывать абсолютно.

### Переопределение встроенного имени исключения

[[2.Углубленная работа с декораторами/2_Классы-декораторы/practice from gpt/Исключение/input_errors.py|input_errors.py]]:

```python
class ConnectionError(Exception):
    pass
```

> [!WARNING] Это имя уже занято встроенным исключением
> `ConnectionError` — стандартное исключение Python (подкласс `OSError`). Объявление одноимённого класса в модуле **перекрывает** встроенное имя в этом файле: любой код в модуле, ссылающийся на `ConnectionError`, получит именно этот кастомный класс, а не системный. Если в файле понадобится настоящий `ConnectionError` (например, при реальном сетевом сбое `socket`), для этого нужно явно использовать `builtins.ConnectionError` или переименовать свой класс.

### Рабочая версия без недостижимого кода — [[2.Углубленная работа с декораторами/2_Классы-декораторы/practice from gpt/Исключение/root.py|root.py]]

```python
def safe_divide(a, b):
    if b == 0:
        raise ValueError("Cannot divide by zero")
    return a / b

while True:
    try:
        a = int(input("a: "))
        b = int(input("b: "))
        print(safe_divide(a, b))
    except ValueError as e:
        print("Error:", e)
```

В отличие от [[2.Углубленная работа с декораторами/2_Классы-декораторы/practice from gpt/Исключение/avg.py|avg.py]], здесь `except ValueError`
ловит сразу оба реальных источника ошибки в этом цикле - и `int(input(...))` на нечисловой ввод, и собственный `raise ValueError` из `safe_divide` при делении на ноль,
без недостижимой ветки `isinstance`.
