# Вложенные декораторы — конспект

Разбор декораторов с `functools.wraps`, декораторов с необязательными аргументами (паттерн `_func=None`) и их сочетания в [[2.Углубленная работа с декораторами/1_Вложенные декораторы/main.py|main.py]].

## 1. Зачем нужен `functools.wraps`

Без `@wraps(func)` обёртка теряет `__name__`, `__doc__` и прочие метаданные исходной функции — это будет `wrapped` вместо оригинального имени, что ломает интроспекцию, логирование и документацию.

Правильный, рабочий вариант из [[2.Углубленная работа с декораторами/1_Вложенные декораторы/practice_1.py|practice_1.py]]:

```python
def greet(func):
    @wraps(func)
    def wrapped():
        print("Hello")
        func()
    return wrapped

@greet
def my_func():
    print("Hi let's play tennis!!!!!!!!")
```

> [!WARNING] `wraps(func)` без `@` ничего не делает
> В [[2.Углубленная работа с декораторами/1_Вложенные декораторы/practice_2.py|practice_2.py]] (`reap`) вызов `wraps(func)` стоит **без декоратора `@` перед ним**:
> ```python
> def reap(func):
>     wraps(func)          # результат никуда не применяется - это просто дохлый вызов
>     def wrapped():
>         func()
>         func()
>     return wrapped
> ```
> `wraps` — это фабрика декоратора: `wraps(func)` возвращает декорирующую функцию, которую нужно **применить** к `wrapped` через `@wraps(func)`. Без `@` вызов не имеет побочного эффекта, и `wrapped.__name__` останется `"wrapped"`.

## 2. Замыкания и состояние декоратора

[[2.Углубленная работа с декораторами/1_Вложенные декораторы/practice_5.py|practice_5.py]] — счётчик вызовов через `nonlocal`:

```python
def count_calls(fun):
    count = 0
    @wraps(fun)
    def wrapped(*args, **kwargs):
        nonlocal count
        count += 1
        print(f"{fun(*args, **kwargs)} ({count})")
    return wrapped
```

`count` живёт в замыкании `wrapped` и переживает между вызовами `say_hi()` — именно поэтому нужен `nonlocal`, иначе `count += 1` создало бы локальную переменную и упало с `UnboundLocalError`.

`main.py` хранит состояние иначе — прямо на атрибуте функции (`func.calls`), что тоже работает, т.к. функции в Python — обычные объекты, которым можно присваивать атрибуты.

> [!WARNING] Баг в `practice_4.py` (`only_positive`)
> ```python
> def only_positive(fun):
>     @wraps(fun)
>     def wrapped(fun):        # <- аргумент называется так же, как декорируемая функция!
>         if fun < 0:
>             print(f"{fun} #Only positive numbers!")
>     return wrapped
> ```
> Параметр `wrapped(fun)` **затеняет** внешнюю `fun` (саму декорируемую функцию) внутри своей области видимости. В итоге `print_age(10)` не печатает `"Your age is 10"` — исходная функция `print_age` вообще ни разу не вызывается, потому что ветка `if fun < 0` не выполняется, а `else`-ветки с вызовом оригинала попросту нет. Комментарий `# Your age is 10` в коде не соответствует реальному выводу.

## 3. Декоратор с необязательными скобками (`_func=None`)

В `main.py` декоратор `trace` можно использовать и как `@trace`, и как `@trace(sep="~")`:

```python
def trace(_func=None, *, sep="="):
    def decorator(func):
        func.calls = 0
        @wraps(func)
        def wrapper(*a, **kw):
            func.calls += 1
            print(sep * func.calls, f"-> {func.__name__}(*{a}, **{kw})")
            try:
                return func(*a, **kw)
            finally:
                print(sep * func.calls, f"<- {func.__name__}(*{a}, **{kw})")
                func.calls -= 1
        return wrapper

    if _func is None:
        return decorator      # вызвали как @trace(sep="~") - вернули фабрику
    else:
        return decorator(_func)   # вызвали как @trace - сразу применили
```

Идея: если декоратор применили без вызова (`@trace`), Python сразу передаёт декорируемую функцию в `_func`, и код сам вызывает `decorator(_func)`. Если декоратор вызвали с аргументами (`@trace(sep="~")`), `_func` остаётся `None`, и наружу возвращается `decorator`, который Python применит на следующем шаге.

## 4. Стек декораторов и порядок применения

```python
@cached
@trace(sep="~")
def fib(n):
    if n < 2:
        return n
    return fib(n - 1) + fib(n - 2)
```

Декораторы применяются снизу вверх: сначала `trace(sep="~")` оборачивает `fib`, затем `cached` оборачивает результат. Итоговое имя `fib` в модуле указывает на `cached`-обёртку.

> [!NOTE] Почему трассировка не печатается на каждый рекурсивный вызов
> Внутри тела `fib` рекурсивные вызовы `fib(n - 1)` обращаются к глобальному имени `fib` — то есть к уже полностью обёрнутой (`cached(trace(...))`) версии. Поэтому при повторном `n` сработает кэш `cached` **раньше**, чем дело дойдёт до `trace`, и строка вызова не напечатается второй раз для одного и того же `n`.

## 5. Декораторы и `multiprocessing`

```python
def multiprocessing_fib(numbers):
    with Pool() as pool:
        results = pool.map(fib, numbers)
    return results
```

> [!WARNING] Кэш не переживает переход в другой процесс
> `results = {}` внутри `cached` — это обычный словарь в памяти одного процесса. `Pool()` в `multiprocessing` поднимает отдельные процессы (на Windows — через `spawn`, с повторным импортом модуля), и у каждого воркера будет **свой собственный, пустой** словарь `results`. Смысл кэширования между вызовами `pool.map` теряется — ускорение даёт только то, что кэш работает *внутри* рекурсии одного вызова `fib(n)` в рамках одного процесса.
