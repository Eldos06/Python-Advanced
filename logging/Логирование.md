# Логирование (`logging`)

От "почему не `print`" к настройке форматтеров, лог-адаптерам для `{}`-стиля и правильному логированию исключений.

## 1. Почему не `print` — [[logging/Suren/1.py|Suren/1.py]]

```python
def process_payment(amount, user_id):
    print(f"Processing payment for user {user_id}")  # нет времени, нет уровня важности, нельзя отключить в prod
    ...
    except Exception as e:
        print(f"Error: {e}")  # нет stacktrace - видно ТОЛЬКО текст исключения

def process_payment_logged(amount, user_id):
    logger.info("Processing payment for user %s", user_id)
    ...
    except Exception as e:
        logger.exception("Error occurred")  # logger.exception печатает ПОЛНЫЙ traceback автоматически
```

`logger.exception(msg)` - это `logger.error(msg, exc_info=True)`: можно вызывать только внутри `except`, сам
допечатывает traceback пойманного исключения к сообщению.

## 2. Конфигурация форматтера — [[logging/Suren/common.py|Suren/common.py]] (и аналогичный [[logging/practice/common.py|practice/common.py]])

```python
def configure_logging(level=logging.info):   # см. предупреждение ниже
    logging.basicConfig(
        level=level,
        datefmt="%Y-%m-%d %H-%M-%S",
        format="[%(asctime)s.%(msecs)03d] %(module)5s:%(lineno)-3d %(levelname)-7s - %(message)s",
    )
```

`%(module)5s:%(lineno)-3d` - выравнивание колонок в логах (модуль вправо на 5 символов, номер строки влево на 3) -
чисто косметика, чтобы строки лога были визуально ровными в консоли.

> [!WARNING] `logging.info` вместо `logging.INFO`
> И тут (`level=logging.info` по умолчанию), и в [[logging/documantation/main.py|documantation/main.py]]
> (`configure_logging(level=logging.info)`) вместо константы `logging.INFO` (число `20`) передаётся сама
> **функция** `logging.info` (шорткат для `root.info(...)`). `basicConfig(level=<функция>)` не упадёт сразу, но
> уровень логирования будет выставлен некорректно - это опечатка на регистре (`INFO` vs `info`), а не разные вещи.

## 3. `%s`-аргументы логируются лениво... но не всегда — [[logging/Suren/2.py|Suren/2.py]] + [[logging/Suren/utils.py|Suren/utils.py]]

```python
logger.debug("Prepare to do something, number: %s, word: %s, user: %s", number, word, user)
```

Смысл `%s` вместо f-строки: если уровень `DEBUG` отключён, `logging` вообще не станет тратить время на
форматирование строки (`str(user)` не вызовется). Но это работает только для **самого форматирования**, не для
вычисления аргументов:

```python
logger.warning("Expensive message: %s", something_expensive())  # sleep(2) внутри
```

> [!WARNING] Аргумент вызывается ДО того, как logger решит, логировать ли вообще
> `something_expensive()` - это обычный вызов функции в Python, он выполнится, даже если уровень `WARNING`
> отключён и сообщение никуда не попадёт. Ленивость `%s` экономит на **форматировании строки**, а не на
> **вычислении аргументов** - если аргумент дорогой (I/O, `sleep`, тяжёлый `__str__`), его всё равно нужно
> оборачивать в `if logger.isEnabledFor(logging.WARNING):` вручную.

## 4. Логирование исключений — [[logging/Suren/exc_example.py|Suren/exc_example.py]] и [[logging/practice/task_1.py|practice/task_1.py]]

```python
try:
    1 / 0
except ZeroDivisionError as e:
    zero_dev_err = e

log.debug("Zero division error: %r", zero_dev_err, exc_info=zero_dev_err)  # exc_info=исключение - явно прицепить traceback
```

```python
try:
    return a / b
except ZeroDivisionError as e:
    logger.error("Attempted to divide by zero: a=%s, b=%s", a, b)
    raise ZeroDivisionError("A very specific bad thing happened.") from e  # `from e` сохраняет исходную причину в traceback
```

`raise ... from e` явно связывает новое исключение со старым (`__cause__`) - в traceback будет видно "во время
обработки этого исключения возникло другое", а не потерянный контекст.

> [!WARNING] Несовпадение сигнатуры в `exc_example.py`
> `def get_weather() -> dict | None:` объявлена **без параметров**, но `main()` зовёт её как `get_weather("Almaty")` -
> это `TypeError: get_weather() takes 0 positional arguments but 1 was given` при реальном запуске. Внутри функции
> тело тоже ссылается на необъявленный `city` (используется `know_weather[city.lower()]`, но переменной `city`
> нигде нет - только закомментированный аргумент). Похоже на недописанный черновик.

## 5. Свой `{}`-стиль вместо `%s` — [[logging/Suren/pretty_braces_formatter.py|pretty_braces_formatter.py]]

```python
class BraceMessage:
    def __init__(self, ftm, args):
        self.ftm, self.args = ftm, args
    def __str__(self):                          # вызовется ТОЛЬКО когда сообщение реально форматируется хендлером
        if len(self.args) == 1 and isinstance((kwargs := self.args[0]), dict):
            return self.ftm.format(**kwargs)     # logger.info("User {user}", {"user": obj}) - именованная подстановка
        return self.ftm.format(*self.args)

class BraceStyleAdapter(logging.LoggerAdapter):
    def log(self, level, msg, /, *args, stacklevel=1, **kwargs):
        if self.isEnabledFor(level):
            msg, kwargs = self.process(msg, kwargs)
            self.logger.log(level, BraceMessage(msg, args), **kwargs, stacklevel=stacklevel + 1)
```

Идея: обернуть сообщение в объект с "ленивым" `__str__`, который вызовется логгером только при реальной записи
(в отличие от готовой f-строки, которая форматируется всегда, до вызова `logger.info`). Так можно использовать
`str.format`-стиль (`{user!r}`, именованные плейсхолдеры) вместо `%s`, сохранив выигрыш в производительности от
ленивого форматирования на отключённых уровнях.

## 6. Минимальный пример с раздельными модулями — [[logging/documantation/main.py|documantation/main.py]] + [[logging/documantation/mylib.py|mylib.py]]

```python
# mylib.py
logger = logging.getLogger(__name__)   # логгер называется по имени модуля - "mylib"
def do_something():
    logger.info('Doing something')
```

```python
# main.py
from mylib import do_something
from Suren.common import configure_logging
logger = logging.getLogger(__name__)   # тут логгер называется "__main__" (или "documantation.main")
```

Стандартный идиоматичный паттерн: `logging.getLogger(__name__)` в каждом модуле даёт логи с именем модуля-источника
прямо в формате (`%(module)s`), а настройка (`configure_logging`/`basicConfig`) вызывается один раз, в точке входа.

## 7. Практика с файлами — [[logging/practice/reader.py|reader.py]]

```python
def count_lines_for_loop(filepath):
    try:
        ...
        logger.info(f"lines are {line_count}")
    except:
        raise FileNotFoundError("File doesn't exist")
```

> [!WARNING] Голый `except:` глотает и переименовывает любую ошибку
> Если `open()` упадёт не из-за отсутствия файла (например, `PermissionError` или `IsADirectoryError`), код всё
> равно перевыбросит `FileNotFoundError` - реальная причина потеряется, а сообщение об ошибке будет вводить в
> заблуждение. Лучше ловить конкретно `FileNotFoundError` (или логировать `logger.exception(...)` с исходным типом).
